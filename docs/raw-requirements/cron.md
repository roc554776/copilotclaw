# Cron 機能 (raw requirement)

<!-- 2026-03-29 -->

## cron 機能の追加

- cron 機能を追加してください
- cron 機能とは、 config に設定すると、定期的に agent を指定したタスクで起動できます。
- ただし、この場合にも cron の度に毎回プレミアムリクエストが消費されるようなことのないようにしてください。最小限に抑えてください。
- cron は複数設定できます。実行間隔と、タスク内容、紐づくチャンネルが指定できます。
- ※ まずは channel に直接紐づく agent のケースだけを実現します。（将来はチャンネルに紐付かない cron も実装します）
- ※ この機能で、直接 client.send することはしません。コストを抑えるためです。

## 設計イメージ

- 具体的には定期的にメッセージを channel に送る機能です。
- cron には名前か id か、とにかく識別子をつけましょう。そうしないと、重複送信を減らしたりする実装もできないですから。
- sender は user でも agent でもなく、 cron 的なのにしましょう。
- この専用 sender からメッセージが入った時、物理セッションが生きていなければ、起動するようにしましょう（いまは sender user のみですが、cron 的なのも対象に追加します。）
- wait tool を修正して、 cron 的な sender のメッセージが来た場合は、そのことを伝えるようにしましょう。
- 障害で、 cron のメッセージが溜まったままになるのはまずいので、未処理のまま stuck している場合は同じメッセージを重複して追加しないようにしましょう。

## cron の enable/disable

- cron は enable/disable で切り替えられるようにする。
- cron の項目ごとに enable/disable を設定できる必要がある。
- 動的に読み込む必要は（まだ）ない。

## chat operator プロンプトの修正

- chat operator のプロンプトの修正をして、 cron 的なのからタスクが来ることがあること、その場合には worker subagent に委譲することをマージしましょう。
- 別件ですが chat operator のプロンプトに、subagent 呼び出しでは常に background mode で、 worker 以外使うな、という内容をマージしておきましょう。

<!-- 2026-03-31 -->
## cron の設定リロードコマンド

- cron の設定をリロードするコマンドを作る
  - cron reload
  - cron list
  - リロード時に、設定差分がないもののタイマーをリセットしないように注意

## cron の設定フィールドの変更

- cron の設定フィールドは、 enabled ではなく disabled にする。（default false）
  - 理由: デフォルトが true のフラグは基本的にバグの温床になる

## archive されたチャンネルの cron

- archive されたチャンネルの cron は設定値に関係なく disabled として扱う
