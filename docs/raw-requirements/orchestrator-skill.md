ただ、このまま適用できる品質ではないです。かなり品質が低いです。

明らかな考慮漏れの例: cross-cutting ルールを CLAUDE.md に書くと subagent もそれに従ってしまうため、無限ループしてしまう。

まずは、既存のものを壊すのではなく、新規にフレームワークを作り、後でそのフレームワークに既存のものを移行していく形がよいと思います。

- いわゆる Orchestrator 的な部分を skill 化する
  - 最初は orchestrator skill をスラッシュコマンドとして呼び出す運用とする
  - 後で agent 化する、もっと洗練された運用をしてもいいが、今は単純な方法を選ぶ。（単に試しやすいから）
  - subagent を呼び出す場合の subagent への指示に固定の表現を入れることで、 subaegnt が Orchestrator skill を呼びださないようにする。
  - subagent の細かい呼び出し workflow は {{skill}}/assets/workflows/*.md に置くことにしましょう。
    - つまり、一旦は skill が定義するビルトイン workflow だけを使う構成にします。将来 pluggable にすればいい。
  - デフォルトの workflow を用意しておきましょう。
    - subagent による作業に対して、常に別の subagent がレビュー、修正、さらに少しでも修正があればさらにレビュー、というループを常に徹底する
    - さらに human の指示から human の期待値を完全に充たせているかチェックし、それが充たせていない場合は、さらに作業ループを回す
  - 明示されなければ、デフォルトの workflow を回す、という運用にしましょう。
- subagent へのプロンプトの基本的なフォーマットと、ルールを定める
  - subagent が読むべき基本的なルールを {{skill}}/agents/worker.md に書いておきます。
    - <xxx /> や <xxx>{{content}}</xxx> のようなタグで変数挿入できるようにします。
      - 2 つの違いはデフォルト値の有無と考えられます。
    - 指示の中で値が与えられる場合は、その値を採用して指示を解釈します。
    - worker は human の指示原文から期待値を読み取り、 Orchestrator からの指示が本当に期待値を充たせるものなのか判断し、 Orchestrator の指示ミスの可能性を常に考慮するようにします。
    - 一旦 worker は subagent の呼び出しを禁止としましょう。
    - worker の Orchestrator への response のフォーマットも定めておきましょう。
      - 例えば編集したファイルのリストや、見付けた問題点（スコープ外でも）、指示を完遂できたのか、指示ミスと考えて指示を再解釈した部分、などを含めさせるようにします。
  - subagent への指示は、 「 {{skill}}/agents/worker.md を読み、従え。ただし変数部は以下を使うこと...」のようにします。全ての tool を使える、 generic な agent を必ず使わせましょう。
- 全体として、例えば opus や sonnet の指定など、パラメトライズするべき部分の抽象化が必要
  - これは一旦パラメタの抽出だけして、ハードコードすればよい（要は、ハードコードはしているが、マジックナンバーにはなっていないという状態までは持っていく）

一旦ここまでを skill-creator skill を使って実装しましょう。
