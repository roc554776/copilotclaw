- **Gateway**: singleton daemon (default port 19741, configurable via config file or COPILOTCLAW_PORT env var), manages channels, inputs, and messages; reports GATEWAY_VERSION (from package.json), agentCompatibility, profile, and config (model, stateDir, workspaceRoot, auth.github, otel, debug) via /api/status; proxies Copilot quota and models from agent via /api/quota and /api/models (v0.61.0: both endpoints also include GitHub API data вҖ” githubUsage and githubModels respectively, fetched via github-api.ts); serves recent logs via /api/logs (ring buffer); hosts observability infrastructure (session event store, system prompt snapshots, status page, events page); initializes OTel at startup (logs + metrics export via OTLP HTTP) and shuts down on exit
- **Agent**: single process, executes physical sessions on behalf of gateway; communicates with gateway exclusively via IPC (stream for push-based messaging, short-lived connections for status/stop/quota/models); sendToGateway buffers messages to in-memory queue + JSONL file when stream disconnected (maxQueueSize configurable via gateway config, default 10000), queue restored from disk on init via initSendQueue(dataDir) (v0.82.0: dispatches Initialized event to reduceSendQueue), flushed on stream connect via flushSendQueue() with ACK protocol (v0.79.0: flush assigns _queueId, waits for message_acknowledged IPC from gateway before clearing disk; v0.82.0: reduceSendQueue manages flushInProgress, pendingAckIds, ClearDisk command); receives OTel config from gateway via IPC stream config push and initializes its own OTel setup independently; does not manage abstract sessions, channel bindings, backoff, or persistence (moved to gateway SessionOrchestrator in v0.49.0)
- **Agent Session**: wraps a Copilot SDK session with its own sessionId, optionally bound to a channel
- **ChannelProvider**: plugin interface for chat mediums (built-in chat, Discord, Telegram, etc.); providers handle medium-specific routes and receive message notifications
- **BuiltinChatChannel**: default ChannelProvider вҖ” serves dashboard UI at "/", channel SSE at "/api/events" (channelId required), global SSE at "/api/global-events", broadcasts via SseBroadcaster

## Process Model

- Gateway: CLI spawns daemon (detached), CLI checks /api/status agentCompatibility after healthy, exits 1 on incompatible
- Agent: single process, singleton via Unix domain socket (`copilotclaw-agent.sock`, or `copilotclaw-agent-{{profile}}.sock` when profiled)
- Gateway start вҶ’ agent process ensure: IPC status + version check, spawn if absent; agent status response includes bootId (UUID, unique per process start)
- Gateway daemon вҶ’ periodic agent monitor (30s interval): re-runs ensureAgent, logs failures, recovers automatically; max 3 consecutive failures before error-level logging; detects agent status changes (agentCompatibility, agentVersion) and broadcasts `agent_status_change` / `agent_compatibility_change` events via broadcastGlobal to all global SSE subscribers
- Force-restart flow: ensureAgent returns old bootId on force-restart вҶ’ daemon calls waitForNewAgent to poll until different bootId appears before proceeding
- Gateway stop вҶ’ gateway only (agent process NOT stopped)
- Gateway restart вҶ’ POST /api/stop, wait for port free, then start (restart.ts)
- Agent вҶ” Gateway: IPC stream (v0.35.0) вҖ” persistent bidirectional connection; gateway pushes config and agent_notify; agent pushes channel messages, session events, system prompts; agent sends request-response for drain/peek/flush/list_messages, hook RPCs (SDK hooks forwarded to gateway for centralized processing), and lifecycle RPCs (agent queries gateway for session lifecycle action)
- Gateway SessionOrchestrator manages abstract sessions (channel bindings, suspend/revive, backoff, max age); sends start_physical_session/stop_physical_session commands to agent via IPC stream
- Agent listens for start_physical_session/stop_physical_session from gateway; sends physical_session_started/physical_session_ended back to gateway
- User message POST triggers SessionController.deliverMessage() which checks orchestrator and starts physical session via agent (push-based, no polling)

## Session Keepalive

- `copilotclaw_wait` tool blocks for up to 25 min polling gateway for input (keepalive timeout)
- Tool execution keeps Copilot SDK session active (CLI idle timeout = 30 min)
- On keepalive timeout: tool returns empty вҶ’ keepalive instruction вҶ’ LLM re-invokes tool
- Premium request consumption: ~1 per 30 min (idle), plus 1 per user interaction cycle

## Subagent Completion Notification (v0.16.0+, gateway-handled v0.43.0)

- SDK events `subagent.completed` and `subagent.failed` forwarded from agent to gateway via IPC stream session_event (with sessionId)
- Gateway detects direct subagent calls (no parentToolCallId in event data), looks up channelId from orchestrator, and inserts `[SUBAGENT COMPLETED/FAILED]` system message into the channel's pending queue, then sends agent_notify
- Agent receives system messages via normal pending drain in `copilotclaw_wait`; combineMessages joins messages as-is (no sender-type interpretation); message formatting (e.g., `[CRON TASK]`, `[SYSTEM EVENT]` prefixes) is gateway's responsibility (daemon.ts onToolCall handler)
- Agent-side status tracking retained (subagent session status updated on events) for dashboard display only

## Session Lifecycle (v0.18.0: Persistent Channel Bindings, v0.49.0: Gateway-Side Orchestration, v0.64.0: SessionController, v0.80.0-v0.82.0: Reducer Architecture)

- **Reducer Architecture (v0.80.0-v0.82.0)**: All session state mutations go through pure reducer functions. Gateway: `reduceAbstractSession` (session-reducer.ts, 47 unit tests), `reduceChannel` (channel-reducer.ts), `reducePendingQueue` (pending-queue-reducer.ts), `reduceChannelSse` / `reduceGlobalSse` (sse-broadcaster-reducer.ts), `reduceEventBus` (event-bus.ts). Agent: `reducePhysicalSession` / `reduceCopilotClient` (session-reducer.ts, 26 unit tests), `reduceSendQueue` / `reduceRpc` (ipc-reducers.ts). Effect runtime (effect-runtime.ts) executes commands. `applyWorldState()` is the sole write path for AbstractSession state.
- **SessionController (v0.64.0, updated v0.80.0-v0.81.0)**: Centralizes all session lifecycle transitions and message delivery in a single class (session-controller.ts). All session status changes MUST go through `dispatchEvent()` вҶ’ `reduceAbstractSession()` вҶ’ `applyWorldState()` (v0.81.0: VALID_TRANSITIONS map, transition(), and broadcastStatusChange() private methods deleted). Maintains per-session ephemeral context (pendingReplyExpected, reminderState, lastIdleHasBackgroundTasks) replacing scattered daemon.ts Maps. Single entry point deliverMessage() for all incoming messages. Delegates persistence and binding management to SessionOrchestrator internally.
- **Responsibility Split (v0.49.0, v0.64.0)**: SessionController owns lifecycle transitions and message delivery; SessionOrchestrator owns abstract session persistence (channel bindings, suspend/revive, backoff, max age, SQLite); agent only executes physical sessions (Copilot SDK sessions) on gateway command
- **Abstract vs. Physical Sessions**: Abstract session (sessionId, bound to channel) is separate from physical session (Copilot SDK session). When physical session ends unexpectedly, abstract session transitions to "suspended" (not deleted), preserving channel binding.
- **Session Status**: "new" вҶ’ "starting" вҶ’ "waiting" вҶ’ "notified" вҶ’ "processing" вҶ’ "idle" вҶ’ "suspended" or "stopped"
  - "new": initial status on startSession creation (before physical session starts)
  - "starting": physical session initializing
  - "waiting": idle, awaiting user input (keepalive tool polling gateway)
  - "notified": message arrived on a waiting session (set by SessionController on deliverMessage for waiting sessions)
  - "processing": handling LLM requests or non-wait tool calls
  - "idle": physical session ended normally (idleSession); abstract session preserved but not active; excluded from hasActiveSessionForChannel, idleAllActive, reconcileWithAgent
  - "suspended": physical session stopped unexpectedly or max age reached; abstract session preserved for revival (physicalSessionId retained)
  - "stopped": explicit stopSession() вҖ” fully removes abstract session and channel binding
- **Gateway-Driven Session Commands (v0.49.0)**: Gateway sends start_physical_session (with sessionId, optional physicalSessionId (renamed from copilotSessionId v0.68.0), optional model; channelId removed v0.55.0), stop_physical_session (with sessionId), and disconnect_physical_session (with sessionId, v0.68.0) to agent via IPC stream; agent accepts both physicalSessionId and copilotSessionId (legacy) in start_physical_session; agent sends physical_session_started (sessionId, copilotSessionId, model) and physical_session_ended (sessionId, reason, copilotSessionId, elapsedMs, tokens, error) back to gateway
- **Suspension via daemon periodic check**: gateway daemon checks all sessions periodically (30s); maxAge check suspends sessions exceeding 2 days (default, configurable) by dispatching MaxAgeExceeded event via SessionController вҶ’ reduceAbstractSession() вҶ’ StopPhysicalSession command (v0.80.0+; previously orchestrator.suspendSession() direct call)
- **Revival via orchestrator.startSession**: when a message arrives for a channel with a suspended, idle, or new session, orchestrator revives (sets status to "starting") and gateway sends start_physical_session with preserved physicalSessionId; agent-side resumeSession wrapped in try/catch вҖ” on failure, clears physicalSessionId and falls back to createSession (v0.38.0)
- **Stream disconnect handling**: onStreamDisconnected calls orchestrator.idleAllActive() to idle all active sessions (agent restart scenario вҖ” physical sessions are gone but abstract sessions remain visible with last state); onStreamConnected waits for agent's running_sessions report to reconcile orchestrator state before starting new sessions; agent flushes send queue (buffered messages from disconnected period) before sending running_sessions report on stream connect
- **Running sessions reconciliation (v0.50.0)**: Agent sends running_sessions report on stream connect (listing all non-suspended physical sessions with sessionId, channelId, status); gateway daemon passes to orchestrator.reconcileWithAgent() which idles stale sessions not reported by agent, revives suspended/idle sessions that are still alive in agent; checkAllChannelsPending runs after reconciliation
- **Binding Persistence (v0.49.0, updated v0.79.0)**: SessionOrchestrator uses SQLite (session-orchestrator.db in data dir, WAL mode); abstract_sessions table (sessionId PK, channelId, status, startedAt, physicalSessionId (renamed from copilotSessionId v0.79.0 schema v3вҶ’v4), cumulativeInputTokens, cumulativeOutputTokens, physicalSessionHistory JSON); orchestrator_schema_version table tracks migration version; persistSession() upserts on every mutation; loadFromDb() on construction (restores physicalSession from history for idle sessions so they remain visible in UI); schema migration system (LATEST_SCHEMA_VERSION=4, SCHEMA_MIGRATIONS registry): initSchema creates base tables, runDataMigrations applies sequential migrations (v0вҶ’v1: suspended+channel+history вҶ’ idle; v1вҶ’v2: remaining non-idle+channel вҶ’ idle if history exists, вҶ’ new if no history; v2вҶ’v3: fix idle+no-history вҶ’ new; v3вҶ’v4: ALTER TABLE abstract_sessions RENAME COLUMN copilotSessionId TO physicalSessionId); legacy one-time migration from agent-bindings.json (renames to .migrated after migration)
- **Cumulative Token Tracking (v0.27.0)**: SessionOrchestrator tracks cumulativeInputTokens and cumulativeOutputTokens per abstract session; suspendSession() accumulates tokens from physical session before clearing; gateway updates physical session via onPhysicalSessionEnded handler; dashboard shows cumulative tokens
- **Stopped Session History (v0.30.0)**: SessionOrchestrator maintains physicalSessionHistory (PhysicalSessionSummary[]) per abstract session; suspendSession() pushes physical session to history before clearing; capped at 10 entries (oldest removed); persisted in SQLite
- **Channel backoff (v0.82.0: persisted)**: SessionOrchestrator tracks channelBackoff map; persisted to store.db channel_backoff table (schema v6вҶ’v7); loaded from DB on construction so backoff survives gateway restarts; daemon records backoff on rapid failure (onPhysicalSessionEnded checks elapsedMs < rapidFailureThresholdMs); isChannelInBackoff() checked before starting sessions
- **Lifecycle RPC (v0.52.0, v0.64.0: SessionController)**: Agent queries gateway via lifecycle request-response RPC (queryLifecycleAction) when session reaches idle or error state; gateway returns action: "stop" (agent calls client.stop()), "reinject" (agent re-enters session loop), or "wait" (keep session alive); default when gateway is offline = "wait" (keep session alive); gateway daemon delegates to SessionController.decideLifecycleAction() which returns "stop" + clearCopilotSessionId on error; on idle checks hasBackgroundTasks from onSessionIdle (true вҶ’ "wait", subagent still running scenario) then copilotclaw_wait active state (вҶ’ "wait"), otherwise "stop" (true idle, no background tasks)
- **Disconnect vs. Stop (v0.68.0)**: disconnectPhysicalSession aborts the session and calls session.disconnect() without client.stop(), preserving physicalSessionId for resume; stopPhysicalSession aborts + session.disconnect() + entry removal (no client.stop() вҖ” singleton client shared across sessions); stopAllPhysicalSessions stops the singleton client at the end; idleSession uses disconnectPhysicalSession (lightweight teardown, enables faster resume)
- **Channel notifications**: gateway daemon inserts system messages on unexpected physical session stop (with error detail) and flushes pending messages for the channel

## Key Constraints

- Gateway and agent are independent processes (gateway stop does NOT stop agent)
- Startup direction: always gateway вҶ’ agent (agent never starts gateway)
- Agent process ensure: gateway start time only (NOT on user message POST)
- Agent session ensure: gateway responsibility via SessionOrchestrator (sends start_physical_session/stop_physical_session to agent via IPC stream)
- Agent version check: gateway enforces minimum agent version (MIN_AGENT_VERSION=0.79.0, exported from agent-manager.ts) at start; force-restart on mismatch with reconnectStream(); checkCompatibility()/getMinAgentVersion() expose compatibility status; CLI checkAgentCompatibility polls /api/status when waitForAgent=true (used after force-restart to wait for new agent bootId)
- Log capture: daemon creates LogBuffer (ring buffer), intercepts console via interceptConsole(); logs served at /api/logs (snapshot) and pushed in real-time via global SSE log_appended events (v0.73.0: daemon installs setOnAppend hook on LogBuffer after startServer; dashboard logs panel subscribes via global SSE instead of polling); LogBuffer optionally writes structured JSON lines to file via enableFileOutput() (gateway.log); agent spawned with stderr redirected to agent.log; agent process initializes its own StructuredLogger writing to agent.log
- Structured logging: StructuredLogger (intentionally duplicated in gateway and agent packages) writes JSON Lines (StructuredLogEntry: ts, level, component, msg, data?) to file via appendFileSync; bridges to OpenTelemetry via optional OtelLoggerBridge parameter (emits log records to OTel LoggerProvider when configured); agent uses structured JSON fallback pattern (console.error with JSON.stringify) before StructuredLogger is initialized вҖ” applies to index.ts module-level log/logError, PhysicalSessionManager defaultLog/defaultLogError, and channel.ts ChannelToolDeps logError
- OpenTelemetry: OTel setup module (otel.ts, intentionally duplicated in gateway and agent) initializes OTLP HTTP exporters for logs and metrics; gateway additionally defines application-level metrics (otel-metrics.ts: session count gauges, token usage counters); endpoints configured via config.otel.endpoints (empty = noop export); agent receives OTel config from gateway via IPC stream config push and initializes independently with serviceName "copilotclaw-agent"
- Channel backoff: SessionOrchestrator tracks channelBackoff map; persisted to store.db channel_backoff table (v0.82.0 вҖ” schema v6вҶ’v7; persistChannelBackoff / clearChannelBackoff / loadChannelBackoffs); loaded from DB on SessionOrchestrator construction so backoff state survives gateway restarts; gateway daemon records backoff on rapid failure; isChannelInBackoff() checked before starting sessions (prevents retry storms)
- All Copilot SDK dependencies must be mocked in tests вҖ” including E2E. Real Copilot sessions must never be used in automated tests (authentication requirement and BAN risk)
- Test doubles must be implemented in place, never deferred as skip
- Test runners: vitest for unit + E2E (gateway 804 + frontend 84 = 888 gateway+frontend; agent 158; total 1046 vitest), Playwright for browser E2E (27 tests); total 1073; gateway vitest excludes test/browser/ directory; v0.82.0: +47 gateway reducer unit tests, +26 agent reducer unit tests, +regression tests (starting-stuck, processing-deadlock scenarios); v0.83.0: +regression tests (cron notify timing via session-lifecycle-regression.test.ts + session-reducer.test.ts, IPC flush ordering via send-queue.test.ts), +1 Playwright (session_status_change backward compat)
- Frontend tests: vitest + jsdom + @testing-library/react for React SPA component tests (SessionEventsPage, StatusPage, DashboardPage, SessionsListPage, useAutoScroll, MessageAvatar, ProfileModal); v0.78.0 +4 tests: MessageAvatar size prop (Г—2), ProfileModal avatar color via colorFromString (Г—2)
