copilotclaw プロジェクトにおける実装作業の全ワークフロー。以下の手順を順序通りに実行する。

## Phase: 実装

- 指示の内容から、対応する要望を確認する
  - `/process-requirements` で整理済みなら、raw-requirements / requirements / proposal を読んで確認する
  - まだなら `/process-requirements` で整理する
- 実現が必要な範囲を再調査、再確認する
  - 関連するコードを読み、現状を把握する
  - ドキュメントと実装が一致していると盲信しない
- 実装を完遂する
- 自動テストの追加と実施も含む（省略不可）
  - 新しく追加・変更した全てのコードパスに対して、対応するテストが存在することを確認する
  - 「既存テストが通った」だけでは不十分。追加したコードに対応するテストを列挙し、漏れがないか確認する
- ビルド確認（`pnpm run build`）も実施する（省略不可。テスト通過 ≠ ビルド成功）
- ドキュメントの更新も含む（省略不可）
  - 利用者向けの README 等の更新も忘れずに
  - 抽象セッションと物理セッションなど、プロジェクト固有の概念は正確に区別して記述する
- copilotclaw の各パッケージのバージョンを更新する
  - 各パッケージのバージョンは一律に揃える
  - gateway と agent の compatibility が壊れる場合には、最低バージョンを引き上げる
  - compatibility が壊れていない場合には引き上げてはいけない（無駄な cost になる）
- この時点では、未実現のマーカーはドキュメント上に残したままにしなければいけない

## Phase: コミット

- feature ブランチにいることを確認する
  - いますでに default ブランチ（=main）以外のブランチにいる場合は、基本的にそれが feature ブランチ
  - まだ feature ブランチにいないなら、差分にもとづいてブランチを作る
- コミットする。コミットメッセージは短く 1 行
  - NG: `Co-Authored-By: Claude Opus 4.6 (1M context) <noreply@anthropic.com>` みたいなのをつけてはだめ

## Phase: コードマップ更新

- subagent で、update codemap して、コミットする
  - （重要）subagent で実施しないと、記憶が喪失する

## Phase: レビュー

- レビュー & 修正を subagent に依頼して、僅かでも修正があったら、修正後のコードを再度レビュー & 修正することを無制限に繰り返す
  - レビューの際には、コードの品質だけでなく、要望を満たしているかも確認する
  - CRITICAL/HIGH/MEDIUM/LOW のいずれの場合でも、この場で完全に修正し切る
  - 既存の問題に関する修正もこの場で即時、完全に完遂する

## Phase: 実現確認

- 要望と、実現内容をあらためて確認する
  - 要望の中で、実現できなかったものがないか、再度確認する
  - 実現できたものだけ、未実現のマーカーをドキュメント上から削除する
    - その際、実際のコードと、raw-requirements/requirements/proposal をそれぞれ確認して、慎重に結論する
  - もし要望の中で、実現できなかったものがある場合、必ず未実現であることをドキュメント上にマークする

## Phase: 報告

- 基本的に、全ての要望を見たす必要がある
- 要望と、その実現状況を表に整理して示す
- もし要望の中で、実現できなかったものがある場合、その理由も示す
- 部分的または全部が未実現の場合、必ず未実現であることをドキュメント上にマークする

| 要望 | 実現状況 |
|------|----------|
| ... | 実現済み / 未実現（理由） |
